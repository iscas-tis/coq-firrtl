A FIRRTL \emph{circuit} comprises a list of modules with a designated top-level module, each of which representing a hardware block that can
be instantiated. For instance, the FIRRTL circuit shown in Figure~\ref{fig:example} has two modules \prettyll{A} and \prettyll{B}, where the top-level module of the circuit is module \prettyll{A}.
A \emph{module} consists of a list of ports and an sequence of statements.
Each port has an identifier,  an direction either \prettyll{input} or \prettyll{output}, and a data type.
For instance, the module {\tt A} has two input ports \prettyll{clk} and \prettyll{rst} and one output port \prettyll{io}, with
types \prettyll{Clock}, \prettyll{UInt<1>} and  \prettyll{\{flip enable: UInt<1>, calc: UInt<3>\}}, respectively.
Each \emph{statement} defines a wire, register, node, connect (<=), partial connect (<-), conditional (when), or instance (inst of), and so on.
A \emph{type} for specifying the structure of the data, can be either a \emph{ground type}, e.g.  \prettyll{Clock} (carrying clock signal) and \prettyll{UInt<1>} (unsigned integer of bit width 1);
or an \emph{aggregate type}, which is either a \emph{vector type}, e.g.  \prettyll{UInt[3]} (a three element vector of unsigned integers),
or a \emph{bundle type}, e.g. \prettyll{\{flip enable: UInt<1>, calc: UInt<3>\}}  comprising two signals of the ground types  \prettyll{UInt<1>} and  \prettyll{UInt<3>} respectively. A circuit \emph{component} is an element in a FIRRTL circuit that can be referred to by its name or identifier. Therefore, a component can be a port, node, wire, register, or instance, and so on. \zhilin{component: a double check is needed}

The syntactic rules of FIRRTL are given in Figure~\ref{fig-firrtl-synt-simp} in Extended Backus-Naur Form (EBNF), where \{$\cdots$\} denotes repetition \fu{(zero or more times)}
and [$\cdots$] denotes option (\fu{either included or discarded}), moreover, the terminals are doubly quoted \mbox{\tt ``} and \mbox{\tt ''}. We stress that the syntactic rules of FIRRTL presented here is a simplification of those in the official specification\footnote{\url{https://github.com/chipsalliance/firrtl-spec/blob/main/spec.md}}. For instance, indent, dedent and newline terminals are omitted for readability, moreover, the types and constructs for analog signals are also omitted, since we focus on digital circuits in this work. In the syntax, memory is also omitted since its semantics is still under active debate \footnote{\url{https://github.com/chipsalliance/chisel/issues/3112}}.

From the syntax, FIRRTL features the high-level constructs like vector types, bundle types, conditional statements, connects, partial connects, and modules.
These high-level constructs are then gradually removed by a sequence of lowering transformations. During each lowering transformation, the circuit is rewritten into an equivalent circuit using simpler, lower-level
constructs. Eventually the circuit is simplified to its most restricted form, resembling a structured netlist, which allows for easy translation to an output language (e.g. Verilog). This form is given the name lowered FIRRTL (abbreviated as LoFIRRTL) and is a strict subset of the full FIRRTL language. For clarity, the original form is called high FIRRTL (HiFIRRTL). Besides LoFIRRTL, there is another lowered form of FIRRTL, called MidFIRRTL, which resides between HiFIRRTL and LoFIRRTL. Nevertheless, in this work, we do not introduce MidFIRRTL, since it can be seen as an internal representation of FIRRTL that do not interface with the other parts of the Chisel toolchain.


A FIRRTL circuit is a LoFIRRTL circuit if it obeys the following restrictions:
\begin{itemize}
\item All widths must be explicitly defined.
\item The conditional statement is not used.
\item All circuit components must be declared with a ground type.
\item The partial connect statement is not used.
\item All components are connected to at most once.
\end{itemize}
More precisely, a LoFIRRTL circuit is a FIRRTL circuit that is defined by the syntactic rules in Figure~\ref{fig-lofirrtl-synt} and satisfies that in each module, each component is connected to at most once.

\subsection{Lowering Transformations of FIRRTL}

In the sequel, we give an informal introduction to the lowering transformations (aka compilation passes) of  FIRRTL. The dependencies between these compilation passes are illustrated in Figure~\ref{fig-c-pass}.

\paragraph*{InferTypes} The InferTypes pass infers the types of components. For instance, InferTypes infers from the code snippet \prettyll{ wire x: UInt   x <= UInt<5>(15)   x <= UInt<1>(1) } that $x$'s type is \prettyll{ UInt}. Note that for a connect statement   \prettyll{ x <= e}, InferTypes only checks that the type of $e$ is equivalent to that of $x$ (where the bit widths are ignored), but does not update the bit width of $x$ to that of $e$.

\paragraph*{InferWidths}
The InferWidths pass infers the widths of all signals with unknown widths, after InferTypes has been applied. It is a \emph{global} width inference algorithm. Instances of the same module with unknown input port widths will be assigned the largest width of all assignments to each of its instance ports. It infers the smallest width that is larger than all assigned widths to a signal. Note that this means that dummy assignments that are overwritten by last-connect-semantics can still influence width inference. For instance, InferWidths infers from the following code snippet that $x$'s width is $5$, but with an assignment of \prettyll{UInt(1)}: 
\prettyll{  wire x: UInt   x <= UInt<5>(15)   x <= UInt<1>(1)}.

\paragraph*{InferResets} %% that infers concrete resets, i.e. AsyncReset or UInt<1>,
The InferResets pass infers the abstract reset type \prettyll{Reset} to the concrete reset types, i.e. \prettyll{AsyncReset} for asynchronous resets or \prettyll{UInt<1>} for synchronous resets. For example, in the following code with an input signal \prettyll{in} of type \prettyll{UInt<1>} and a wire signal \prettyll{rst} of abstract type \prettyll{Reset}, the \prettyll{rst} signal will be inferred as a synchronous reset based on the type of \prettyll{in}: \prettyll{ input in: UInt<1>   wire rst: Reset   rst <= in }. If an uninferred signal is instantiated with different concrete reset types, such conflict behavior is an error.

\paragraph{ExpandConnects} The ExpandConnects pass expands the connect and partial connect statements, after the types of components have been inferred by InferTypes. For instance, the ExpandConnects pass transforms the following program

\centering
\begin{tabular}{cccc}

module MyModule:
  input x: {a: UInt, b: UInt}
  output y:{a: UInt, b: UInt}
  y <= x
\end{lstlisting} &
into
&

\begin{lstlisting}[basicstyle=\scriptsize\ttfamily]

  input x: {a: UInt, b: UInt}
  output y:{a: UInt, b: UInt}
  y.a <= x.a
  y.b <= x.b
\end{lstlisting}
\end{tabular}
\end{figure}

\paragraph{ExpandWhens} The ExpandWhens pass expands the when statements and removes the last connect semantics, after the connect statements have been expanded by ExpandConnects. For instance, the ExpandWhens pass transforms the following program
\begin{figure}[h]
\centering
\begin{tabular}{cccc}
\begin{lstlisting}[basicstyle=\scriptsize\ttfamily]
module MyModule :
  input in: UInt<1>[2]
  input n: UInt<1>
  output out: UInt<1>
  out <= n
  when eq(n, UInt(0)) :
    out <= in[0]
  else when eq(n, UInt(1)) :
    out <= in[1]
\end{lstlisting}&
into
&
&
\begin{lstlisting}[basicstyle=\scriptsize\ttfamily] 
module MyModule :
  input in: UInt<1>[2]
  input n: UInt<1>
  output out: UInt<1>
  out <= mux(n, in[1], in[0])
\end{lstlisting}
\end{tabular}
\end{figure}

Note that according to the last connect semantics, the connect statement \begin{lstinline} !out <= n! \end{lstinline} is removed by ExpandWhens.

\paragraph{LowerTypes} The LowerTypes pass flattens bundles and vectors, after the connect and partial connect statements have been expanded by ExpandConnects. Nevertheless, LowerTypes does not change the implicit bundles, e.g. module instances. After the application of LowerTypes, a module instance is still a bundle of depth 1 (instance -> port).
