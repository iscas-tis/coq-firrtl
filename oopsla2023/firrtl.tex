\subsection{FIRRTL IR}

Below is an example of FIRRTL programs.
\begin{lstlisting}
circuit MyCircuit:
  module MyModule :
    input in: UInt[3]
    input n: UInt<2>
    output out: {valid: UInt<1>, ready: UInt<1>,  result: UInt<1>}
    when eq(n, UInt(0)) :
      out.valid <= in[0]
    else when eq(n, UInt(1)) :
      out.ready <= in[1]
    else when eq(n, UInt(2)) :
      out.result <= in[2]
\end{lstlisting}
A FIRRTL \emph{circuit} comprises modules, each of which include several ports and statements. 
Each \emph{port} has an identifier and types and is either an input or output. 
A \emph{module} includes ports and  a sequence of statements. 
%A \emph{port}  is either an input port or an output port, and has an identifier as well as a type. 
Each \emph{statement} defines a wire, register, node, connect (<=) or partial connect (<-), or conditional (when), and so on. 
%A \emph{statement} is either a declaration of wire, register, instance, node, or a connection (<=) or partial connection (<-), or a conditional (when) statement. 
A \emph{type} can be either a \emph{ground type}, e.g. \begin{lstinline}!UInt<2>!\end{lstinline} (unsigned integer of bit width 2), or an \emph{aggregate type}, which is either a \emph{vector type}, e.g. \begin{lstinline}!UInt[3]!\end{lstinline} (a three element vector of unsigned integers), or a \emph{bundle type} e.g \begin{lstinline}! {valid: UInt<1>, ready: UInt<1>,  result: UInt<1>} !\end{lstinline} which comprises three signals of the ground type  \begin{lstinline}!UInt<1>!\end{lstinline}. A circuit \emph{component} is an element in a FIRRTL circuit that can be referred to by its name or identifier. Therefore, a component can be a port, node, wire, register, or instance, and so on. \zhilin{component: a double check is needed}
%

The syntactic rules of FIRRTL are given in Figure~\ref{fig-firrtl-synt-simp}. Note that these rules are in Extended Backus–Naur form, where \{· · · \} denotes repetition (possibly empty) and [· · ·] denotes option, moreover, the terminals are doubly quoted. Moreover,  the syntactic rules of FIRRTL presented here is a simplification of those in the official specification\footnote{\url{https://github.com/chipsalliance/firrtl-spec/blob/main/spec.md}}. For instance, indent, dedent and newline terminals are omitted for readability, moreover, the type and constructions for analog signals are also omitted, since we focus on digital circuits in this work. In the syntax, memory is also omitted since its semantics is still under active debate \footnote{\url{https://github.com/chipsalliance/chisel/issues/3112}}.

From the syntax, FIRRTL features the high-level constructs like vector types, bundle types, conditional statements, connects, partial connects, and modules.
These high-level constructs are then gradually removed by a sequence of lowering transformations. During each lowering transformation, the circuit is rewritten into an equivalent circuit using simpler, lower-level
constructs. Eventually the circuit is simplified to its most restricted form, resembling a structured netlist, which allows for easy translation to an output language (e.g. Verilog). This form is given the name lowered FIRRTL (abbreviated as LoFIRRTL) and is a strict subset of the full FIRRTL language. For clarity, the original form is called high FIRRTL (HiFIRRTL). Besides LoFIRRTL, there is another lowered form of FIRRTL, called MidFIRRTL, which resides between HiFIRRTL and LoFIRRTL. Nevertheless, in this work, we do not introduce MidFIRRTL, since it can be seen as an internal representation of FIRRTL that do not interface with the other parts of the Chisel toolchain. 


A FIRRTL circuit is a LoFIRRTL circuit if it obeys the following restrictions. 
\begin{itemize}
\item All widths must be explicitly defined.
\item The conditional statement is not used.
\item All circuit components must be declared with a ground type.
\item The partial connect statement is not used.
\item All components are connected to at most once.
\end{itemize}
More precisely, a LoFIRRTL circuit is a FIRRTL circuit that is defined by the syntactic rules in Figure~\ref{fig-lofirrtl-synt} and satisfies that in each module, each component is connected to at most once.

\subsection{Lowering Transformations of FIRRTL}

In the sequel, we give an informal introduction to the lowering transformations (aka compilation passes) of  FIRRTL. The dependencies between these compilation passes are illustrated in Figure~\ref{fig-c-pass}.

\paragraph*{InferTypes} The InferTypes pass infers the types of components. For instance, InferTypes infers from the code snippet \begin{lstinline}!  wire x: UInt   x <= UInt<5>(15)   x <= UInt<1>(1)! \end{lstinline} that $x$'s type is \begin{lstinline}!UInt!\end{lstinline}. Note that for a connect statement  \begin{lstinline}! x <= e !\end{lstinline}, InferTypes only checks that the type of $e$ is equivalent to that of $x$ (where the bit widths are ignored), but does not update the bit width of $x$ to that of $e$.

\paragraph*{InferWidths} 
The InferWidths pass infers the widths of all signals with unknown widths, after InferTypes has been applied. It is a \emph{global} width inference algorithm. Instances of the same module with unknown input port widths will be assigned the largest width of all assignments to each of its instance ports. It infers the smallest width that is larger than all assigned widths to a signal. Note that this means that dummy assignments that are overwritten by last-connect-semantics can still influence width inference. For instance, InferWidths infers from the following code snippet that $x$'s width is $5$, but with an assignment of \begin{lstinline}!UInt(1)!\end{lstinline}: \begin{lstinline}!  wire x: UInt   x <= UInt<5>(15)   x <= UInt<1>(1)! \end{lstinline}.

\paragraph*{InferResets} that infers concrete resets, i.e. AsyncReset or UInt<1>,

\paragraph{ExpandConnects} The ExpandConnects pass expands the connect and partial connect statements, after the types of components have been inferred by InferTypes. For instance, the ExpandConnects pass transforms the following program
\begin{lstlisting}
module Foo: 
  input x: {a: UInt, b: UInt} 
  output y:{a: UInt, b: UInt} 
  y <= x
\end{lstlisting}
into
\begin{lstlisting}
module Foo: 
  input x: {a: UInt, b: UInt} 
  output y:{a: UInt, b: UInt} 
  y.a <= x.a
  y.b <= x.b
\end{lstlisting}

\paragraph{ExpandWhens} The ExpandWhens pass expands the when statements and removes the last connect semantics, after the connect statements have been expanded by ExpandConnects. For instance, the ExpandWhens pass transforms the following program 
\begin{lstlisting}
module MyModule :
  input in: UInt<1>[2]
  input n: UInt<1>
  output out: UInt<1>
  out <= n
  when eq(n, UInt(0)) :
    out <= in[0]
  else when eq(n, UInt(1)) :
    out <= in[1]
\end{lstlisting}
into
\begin{lstlisting}
module MyModule :
  input in: UInt<1>[2]
  input n: UInt<1>
  output out: UInt<1>
  out <= mux(n, in[1], in[0])
\end{lstlisting}
Note that according to the last connect semantics, the connect statement \begin{lstinline} !out <= n! \end{lstinline} is removed by ExpandWhens.

\paragraph{LowerTypes} The LowerTypes pass flattens bundles and vectors, after the connect and partial connect statements have been expanded by ExpandConnects. Nevertheless, LowerTypes does not change the implicit bundles, e.g. module instances. After the application of LowerTypes, a module instance is still a bundle of depth 1 (instance -> port).
