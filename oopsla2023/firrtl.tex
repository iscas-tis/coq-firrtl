A FIRRTL \emph{circuit} comprises a list of modules with a designated top-level module, each of which represents a hardware block that can
be instantiated. For instance, the FIRRTL circuit shown in Figure~\ref{fig:example} has two modules \prettyll{A} and \prettyll{B},
where the top-level module of the circuit is module \prettyll{A}.
A \emph{module} consists of a set of ports and a sequence of statements.
Each port has an identifier, a direction (either \prettyll{input} or \prettyll{output}), and a data type.
For instance, the module \prettyll{A} has two input ports \prettyll{clk} and \prettyll{rst} and one output port \prettyll{io}, with
types \prettyll{Clock}, \prettyll{UInt<1>} and  \prettyll{\{flip enable: UInt<1>, calc: UInt<3>\}}, respectively.
Each \emph{statement} defines a wire, register, node, connect (\prettyll{<=}), partial connect (\prettyll{<-}), conditional (\prettyll{when}), or instance (\prettyll{inst of}), and so on.
A \emph{type} for specifying the structure of the data can be either a \emph{ground type,} e.g. \prettyll{Clock} (carrying a clock signal) and \prettyll{UInt<1>} (unsigned integer of bit width 1);
or an \emph{aggregate type}, which is either a \emph{vector type,} e.g. \prettyll{UInt[3]} (a three-element vector of unsigned integers),
or a \emph{bundle type,} e.g. \prettyll{\{flip enable: UInt<1>, calc: UInt<3>\}} comprising two signals of the ground types \prettyll{UInt<1>} and \prettyll{UInt<3>} respectively.
A circuit \emph{component} is a declared element in a FIRRTL circuit that can be referred to by its name or identifier.
Therefore, a component can be a port, node, wire, register, or instance.

The only kind of component that is missing from the above list is the memory.}

The syntactic rules of FIRRTL are given in Figure~\ref{fig-firrtl-synt-simp} in Extended Backus-Naur Form (EBNF), where \{$\cdots$\} denotes repetition \fu{(zero or more times)}
and [$\cdots$] denotes option (\fu{either included or discarded}), moreover, the terminals are doubly quoted \mbox{\tt ``} and \mbox{\tt ''}. We stress that the syntactic rules of FIRRTL presented here is a simplification of those in the official specification\footnote{\url{https://github.com/chipsalliance/firrtl-spec/blob/main/spec.md}}.
For instance, indent, dedent and newline terminals are omitted for readability;
moreover, the types and constructs for analog signals are also omitted, since we focus on digital circuits in this work.
In the syntax, memory is also omitted since its semantics is still under active debate.\footnote{\url{https://github.com/chipsalliance/chisel/issues/3112}}

According to the syntax, FIRRTL features high-level constructs like vector types, bundle types, conditional statements, connects, partial connects, and modules.
These high-level constructs are then gradually removed by a sequence of lowering transformations. During each lowering transformation, the circuit is rewritten into an equivalent circuit using simpler, lower-level
constructs. Eventually the circuit is simplified to its most restricted form, resembling a structured netlist,
which allows for easy translation to an output language (e.g. Verilog).
This form is given the name lowered FIRRTL (abbreviated as LoFIRRTL) and is a strict subset of the full FIRRTL language.
For clarity, the original form is called high FIRRTL (HiFIRRTL).
Besides LoFIRRTL, there is another lowered form of FIRRTL, called MidFIRRTL, which resides between HiFIRRTL and LoFIRRTL.
Nevertheless, in this work, we do not introduce MidFIRRTL,
since it can be seen as an internal representation of FIRRTL that does not interface with the other parts of the Chisel toolchain.


A FIRRTL circuit is a LoFIRRTL circuit if it obeys the following restrictions:
\begin{itemize}
\item All widths are explicitly defined.
\item The conditional statement is not used.
\item All circuit components are declared with a ground type.
\item The partial connect statement is not used.
\item All components are connected to at most once.
\end{itemize}
More precisely, a LoFIRRTL circuit is a FIRRTL circuit that is defined by the syntactic rules in Figure~\ref{fig-lofirrtl-synt}
and satisfies that in each module, each component is connected to at most once.

\subsection{Lowering Transformations of FIRRTL}
\label{subsec-lowering-transformations}

In the sequel, we give an informal introduction to the lowering transformations (aka compilation passes) of  FIRRTL. The dependencies between these compilation passes are illustrated in Figure~\ref{fig-c-pass}.

\paragraph*{InferTypes} The InferTypes pass infers the types of components. For instance, InferTypes infers from the code snippet 
\prettyll{ wire x: UInt   x <= UInt<5>(15) x <= UInt<1>(1) } that $x$'s type is \prettyll{ UInt}. Note that for a connect statement   \prettyll{ x <= e}, InferTypes only checks that the type of $e$ is equivalent to that of $x$ (where the bit widths are ignored), but does not update the bit width of $x$ to that of $e$.

\paragraph*{InferWidths}
The InferWidths pass infers the widths of all signals with unknown widths, after InferTypes has been applied. It is a \emph{global} width inference algorithm. Instances of the same module with unknown input port widths will be assigned the largest width of all assignments to each of its instance ports. It infers the smallest width that is larger than all assigned widths to a signal. Note that this means that dummy assignments that are overwritten by last-connect semantics can still influence width inference. The last-connect semantics means that the last connect statement for a component will take effects and all the preceding connect statements for it will be ignored  in the end. For instance, InferWidths infers from the following code snippet that $x$'s width is $5$, but with a value \prettyll{UInt<1>(1)}: \prettyll{wire x: UInt   x <= UInt<5>(15)    x <= UInt<1>(1)}.


\paragraph*{InferResets} %% that infers concrete resets, i.e. AsyncReset or UInt<1>,
The InferResets pass infers the abstract reset type \prettyll{Reset} to the concrete reset types, i.e. \prettyll{AsyncReset} for asynchronous resets or \prettyll{UInt<1>} for synchronous resets. For example, in the following code with an input signal \prettyll{in} of type \prettyll{UInt<1>} and a wire signal \prettyll{rst} of abstract type \prettyll{Reset}, the \prettyll{rst} signal will be inferred as a synchronous reset based on the type of the input port \prettyll{in}: \prettyll{ input in: UInt<1>   wire rst: Reset   rst <= in }. If an uninferred signal is instantiated with different concrete reset types, such conflict behavior is an error.

\paragraph{ExpandConnects} The ExpandConnects pass expands the connect and partial connect statements, after the types of components have been inferred by InferTypes. For instance, the ExpandConnects pass transforms the following program

\centering
\begin{tabular}{cccc}
%[basicstyle=\footnotesize\ttfamily]
\begin{lstlisting}[boxpos=t,aboveskip=0pt]
module MyModule:
  input x: {a: UInt, b: UInt}

  y <= x
\end{lstlisting} &
\raisebox{-6ex}{\qquad into\qquad}

%[basicstyle=\footnotesize\ttfamily]

  input x: {a: UInt, b: UInt}

  y.b <= x.b

\paragraph{ExpandWhens} The ExpandWhens pass mainly replaces \prettyll{when} statements;
this is necessary because a physical circuit cannot contain conditional elements.
It also applies the so-called last-connect semantics.

For every ground-type writable (sub)component, ExpandWhens retains the last connect or \prettyll{is} \prettyll{invalid} statement
that appears under any combination of \prettyll{when} conditions in the module.
If the (sub)component is declared under multiple conditions,
ExpandWhens selects between them using a multiplexer.

For instance, the ExpandWhens pass transforms the following program
\begin{figure}[h]
\centering
\begin{tabular}{cccc}
%[basicstyle=\footnotesize\ttfamily]
\begin{lstlisting}[boxpos=t,aboveskip=0pt,escapeinside={(*}{*)}]
module TestCondition:
  input in: UInt<32>
  input condition: UInt<1>
  output posv: UInt<32>
  output negv: SInt<33>
  posv <= 0 (* \label{lstinline-connect-before-when} *)
  when condition:
    posv <= in (* \label{lstinline-connect-in-when} *)
    negv is invalid (* \label{lstline-is-invalid} *)
  else:
    wire a: SInt<33> (* \label{lstline-wire-under-when} *)
    a <= neg in
    negv <= a
\end{lstlisting}&
\raisebox{-14ex}{\qquad into\qquad}
&
&
%[basicstyle=\footnotesize\ttfamily]
\begin{lstlisting}[boxpos=t,aboveskip=0pt]
module TestCondition:
  input in: UInt<32>
  input condition: UInt<1>
  output posv: UInt<32>
  output negv: SInt<33>
  wire a: SInt<33>
  posv <= mux(condition, in, 0)
  a <= neg in
  negv <= validif(not condition, a)
\end{lstlisting}
\end{tabular}
\end{figure}

\noindent
Note that according to the last connect semantics,
the connect statement in Line~\ref{lstinline-connect-before-when} of the left listing
is overriden by the one in Line~\ref{lstinline-connect-in-when}
if \prettyll{condition} is true.
Because of this overriding,
ExpandWhens can only run after the connect statements have been expanded by ExpandConnects.

If, under some combination of conditions, the (sub)component is declared but is not connected,
the compiler assumes that the programmer has forgotten a connect statement
and produces an error.
This error can be suppressed by an \prettyll{is invalid} statement
(Line~\ref{lstline-is-invalid} of the left listing),
which explicitly indicates that the connect is missing on purpose,
e.g. because the result is not used or is irrelevant under some conditions.
The multiplexer is replaced by a \prettyll{validif} expression.
%, which may allow later optimization passes to remove superfluous calculations.

If a component is only declared under some condition
(e.g. \prettyll{wire a} in Line~\ref{lstline-wire-under-when} of the left listing),
it is nevertheless included in the LoFIRRTL result,
and the condition is ignored; no multiplexer is inserted upon connections.

\paragraph{LowerTypes} The LowerTypes pass flattens bundles and vectors, after the connect and partial connect statements have been expanded by ExpandConnects. Nevertheless, LowerTypes does not change the implicit bundles, e.g. module instances. After the application of LowerTypes, a module instance is still a bundle of depth 1 (instance -> port).
