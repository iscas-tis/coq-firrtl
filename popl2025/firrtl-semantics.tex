In this section, we define the formal semantics of {\hifirrtl} programs. Since a {\hifirrtl} program provides a high-level description of low-level RTL circuits, %the main idea of 
the formal semantics is essentially to specify which low-level RTL circuits conform to a given {\hifirrtl} program. 

Technically, a {\hifirrtl} program is %\emph{nondeterministic} 
a under-specification in the sense that bit widths of some of its components are not determined. As a result, the formal semantics of a given {\hifirrtl} program is defined as the set of (low-level) {\lofirrtl} programs that conform to it, where all components are of ground types, whose bit widths are all explicit given, there are no \prettyll{when} statements, and each component is connected at most once. 

We shall represent {\lofirrtl} programs %by a data structure called \emph{module graphs}. A module graph $G = (vm, ct)$ represents a module in {\hifirrtl} programs, 
as a pair $(tm, cm)$ where $tm$ is a \emph{type mapping} that records the typing information of ground-type components in the module and $cm$ is a \emph{connection mapping} that records the connections between these components.  
%
A {\hifirrtl} module %$M$, \emph{the semantics of $M$ is defined as 
is interpreted as the set of such pairs that conform to it. 


Hence, the crux of the semantics definition is to define %a module graph 
that $(tm, cm)$ \emph{conforms to} a {\hifirrtl} module $M$ (denoted by $(tm, cm)\models M$). 
To this end,  %is defined as follows: \tl{to see how to improve this.}

	\item %At first, $vm$ is used to determine 
	we first utilize $tm$ to determine the missing widths of components of $M$ and build a type mapping $tmap$, which records the typing information for all the components in $M$,  (Note that the components of $M$ may not be of ground types. ) 
	%
	\item to decide whether $(tm, cm) \models M$, we utilize $tmap$ and the statements in $M$ to construct a pair $(tm', cm')$, and check whether $(tm, cm) = (tm', cm')$ (If it is the case, then $(tm, cm) \models M$ holds.), moreover, during the construction of $(tm', cm')$, the references to the components in $M$ are checked to be well-scoped with respect to its \prettyll{when} statements. 
\end{itemize}
The main technical difficulty for checking $(tm, cm)\models M$ arises from the gap between $(tm, cm)$ and $M$: while $(tm, cm)$ is low-level, $M$ is high-level and may contain aggregate types (e.g. vectors and bundles) as well as \prettyll{when} statements. As a result, when checking $(tm, cm)\models M$, it is necessary to bridge this gap, in particular, to find a match between the ground types in $tm$ and aggregate types in $M$.

Fix a {\hifirrtl} module $M$ which consists of a sequence of port declarations followed by statements.

Let us introduce some notations.

\paragraph*{Types}
We denote by $\gtypes:= (\{\uint, \sint, \uintimp, \sintimp\} \times \Nat) \cup \{\clock,
\reset, \asyncreset\}$ the set of \emph{ground types}. 
Note that $\uintimp$ and $\sintimp$ are introduced to denote the explicit ground types that are instantiated from the implicit ground types. 
In particular, $\gtypes$ is able to model types of implicit width in $M$. For instance, a type $\uint$ where the width is implicit is captured by $(\uintimp, 0)$. Similarly for $\sint$. 
%Let $\bv=\{0,1\}^+$, i.e., the set of bit vectors.

%For a ground type $t = (\tau, n) \in \{UInt, SInt\} \times \Nat$, $n$ is the \emph{width} of $t$, denoted by $wd(t)$.
%

In the definition of $tm$,  the ground types will be extended with the kinds of components, e.g. input and output ports, wires, etc. As a result, we define $\egtypes$, the set of extended ground types, as %the set of vertex sorts.
\begin{itemize}
\item $(s, t)$ with $s \in \{\inport, \outport, \wire, \node\}$ and $t \in \gtypes$,  
\item or $(\register, t, clk, rsig, rval, b)$ such that $t \in \gtypes$, $clk, rsig, rval \in \exps$ and $b\in \{0,1\}$. (Intuitively, $clk$ represents a clock, $rsig$ represents a reset signal, and $rval$ represents a reset value, and $b$ is a Boolean value that denotes whether the reset is asynchronous or not).  
\end{itemize}
Moreover, for convenience, we assume that $\egtypes$ includes two additional elements, i.e. $\svnone$ and $\svundefined$.
% \tl{the same as before}

Besides ground types, the components in $M$ can be of aggregate types. We use $\ftypes$ to denote these types. More precisely, we define $\ftypes$ to be the set of elements of one of the following forms, 
\begin{itemize}
\item $(\gtype, t)$ such that $t \in \gtypes$, 
%
\item $(\vtype, t[n])$ such that $t \in \ftypes$ and $n \in \Nat$, 
%
\item $(\btype, ff)$ such that $ff$ is a sequence of fields. 
\end{itemize}

On the other hand, since $tm$ records the typing information for ground-type components, we define the identifiers in $tm$ as $(v, m) \in \Nat \times \Nat$, aiming at linking them to the identifiers in $M$.

Intuitively, an identifier $(v, m)$ in $tm$ is used to refer to the ground-type components that are obtained by flattening the aggregate-type components in $M$. As a result, for an identifier $(v, m)$, 
\begin{itemize}
\item either $v$ is the identifier of a ground-type component and $m = 0$,
%
\item or $v$ is the identifier of a vector-type component in $M$,  and $m$ is an index in the vector,
% 
\item or $v$ is the identifier of a bundle-type component in $M$, and $m$ is the offset of a ground-type field in the component.
\end{itemize}
Let $\gidset_M$ denote the set of such identifiers $(v,m)$. 

For a bundle-type component, the offsets of its ground-type fields are obtained by a depth-first search over its fields. For instance, for the bundle type 
\begin{center}
\prettyll{output a: \{flip b: \{d: UInt<1>, e: UInt<2>\}, c: UInt<2>\}}, 
\end{center}
%
the ground-type fields in it are \prettyll{a}, \prettyll{c}, \prettyll{d}, \prettyll{e}, and a depth-first search gives us an order $a, d, e, c$. As a result, the offsets of $a, d, e, c$ are $0, 1, 2, 3$ respectively. Note that during the depth-first search, the fields in the same level are traversed according to the order of occurrences in $M$.
