\subsubsection{Formalization of InferWidths}
%By Keyin
We formalize the ``InferWidths'' pass by defining a Coq function {\tt InferWidths\_m}, which contains two process. The first process {\tt InferWidths\_stage1} iterates over the topological sorted order of module's implicit widths ground type components and updates the type map, so that the bit widths of all the components with undefined bit widths can be determined. Then the second process {\tt InferWidths\_stage2} produce a new module with inferred bit widths and explicit widths instead of implicit widths in the original module.

The rationale behind the {\tt InferWidth\_m} function is stated as follows: The function  {\tt InferWidths\_stage1} always chooses the maximum width of the expressions that are connected to the component $x$. 

The behavior of the function {\tt inferwidths\_m} on a module contains the folowing steps :\\
Record the original type map and the connection map according to ports sequecnse and statement sequence in the module. The Coq function {\tt ports\_stmts\_tmap}, given ports list and statements list in the module, generates two corresponding maps, maps from aggregate level components to types and ground type level components to expressions. Note that aggregate connections are not expanded yet so that flip field should be considered when generating the connection map.

For a statement that declares a component $v$ with ftype \{flip a : UInt, b : SInt\},
the {\tt ports\_stmts\_tmap'} function will update $tmap$ by
 $$ {tmap':= tmap \uplus \{v : \{flip \  a : UInt, b : SInt\}\}} $$
\\For a statement that connects $v$ to another component $w$ with ftype \{ flip a : UInt, b : SInt \},
the {\tt ports\_stmts\_tmap'} function will update $var2exprs$ by
 $$ {var2exprs':= var2exprs \uplus \{v.a : \{w.a\} \} \uplus \{v.b : \{w.b\} \}} $$

Generate the topological order of all the uninferred ground type components. Given the vertices set and the graph, the correctness of sorting algorithm {\tt topo_sort} is certified. Because the module components depends on each other through connections and operations, implicit widths should be inferred sequentially in the order of topological order,

Finally, compares the widths of each expression connected to the same component, and chooses the larger width. Since the last connect semantics is not considered, each connection to the component provides a potential width. A function named {\tt max\_ftlist} compares the potential widths and outputs the maximum one. 
the Coq function {\tt InferWidths\_fun} calls {\tt InferWidth_fun} on the list of topological order, where the widths are inferred according to connect statements.

%% recall the semantics of width infering.  There are statements that
%% declare new component, or the statements that define connections. For
%% those components declared with explicit widths, we state that the
%% widths will be kept unchange after the pass.  The implicit width
%% declarations will be collected and await to be assigned to a proper width.
%% The connections give hints to the infering process. For example,
%% \begin{lstlisting}
%%   wire c: SInt
%%   c <= shl(SInt("b-100101100101"), 1)
%% \end{lstlisting}
%% the width of wire $c$ is implicit defined. The connection from an
%% shift-left expression to $c$ will provide the protential width for $c$.
%% According to the rules, the width of c now is 13, because the
%% shift-left expression is of type \lstinline!SInt<13>!. Before applying last
%% connect semantics, the module may contain more than one connections to
%% one target. For example,
%% \begin{lstlisting}
%%   wire c: SInt
%%   wire d: SInt<14>
%%   c <= shl(SInt("b-100101100101"), 1)
%%   c <= d
%% \end{lstlisting}
%% Now, wire $c$ has the width as same as $d$, which is 14. The latter connection
%% from d gives a larger width to the destination $c$.
